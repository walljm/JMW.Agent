# Third-party attribution — IANA Private Enterprise Numbers

`enterprise-numbers.tsv` is a snapshot of IANA's **Private Enterprise Numbers** registry —
the public record of which organization owns each `1.3.6.1.4.1.<number>` OID arc, most
commonly seen as the enterprise number embedded in an SNMP `sysObjectID`
(`1.3.6.1.4.1.<number>...`). It is embedded into the agent binary (`<EmbeddedResource>`) so
vendor resolution works without a network call.

- Source: https://www.iana.org/assignments/enterprise-numbers/enterprise-numbers
- Snapshot date: 2026-07-14 (registry file itself is dated "last updated 2026-07-10")
- Format: `<decimal enterprise number>\t<organization name>`, one per line, sorted by
  number. Rows whose organization name is `Reserved` or `Unassigned` are excluded — a real
  device never reports one of those as its enterprise number, so keeping them out avoids
  ever surfacing "Reserved"/"Unassigned" as a resolved vendor.
- Organization names are IANA's raw registrant strings (mixed casing, legal suffixes,
  occasional odd formatting like `"ciscoSystems"` for enterprise 9) — canonicalized to a
  clean display name downstream by `VendorNormalizer` (`src/Core/Analysis/Normalizers/
  VendorNormalizer.cs`), same as every other vendor-string source in this codebase. This
  file is intentionally NOT pre-cleaned, so it stays a faithful, diffable snapshot of the
  registry and can be regenerated by re-fetching the source URL and re-running the same
  filter (see `docs/plans/vendor-derivation-updates.md` §2.5) without redoing any cleanup.
- No re-refresh mechanism exists — this is a point-in-time snapshot, refreshed manually by
  re-running the fetch. IANA assigns new enterprise numbers continuously, so this file will
  drift over time; re-fetching periodically (or when a specific vendor is missing) keeps it
  current.
- IANA's registry is a public assignment record, not creative content under a software
  license — no license text applies, only source attribution.
